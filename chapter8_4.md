# 问题
muduo的接口设计会明确哪些成员函数是线程安全的，可以跨线程调用；哪些成员函数只能在某个特定线程调用（主要是IO线程）​。为了能在运行时检查这些pre-condition，EventLoop提供了isInLoopThread()和assertInLoopThread()等函数(EventLoop.h L25～L33)，其中用到的EventLoop：​：abortNotInLoopThread()函数的定义从略。

# muduo 线程安全机制分析

## 一、先搞懂几个基础概念

### 1.1 什么是线程？

你可以把一个程序想象成一家餐厅。

- **进程** = 整家餐厅（有厨房、餐桌、收银台等所有资源）
- **线程** = 餐厅里的服务员（多个服务员共享同一家餐厅的资源，各自干各自的活）

一个程序可以同时跑多个线程，就像一家餐厅可以有多个服务员同时工作，效率更高。

### 1.2 什么是线程安全？

多个服务员（线程）**同时操作同一个东西**时，可能会出问题。

> 比如：两个服务员同时去修改同一张桌子的账单，一个要加菜，一个要结账——如果没有协调，账单就乱了。

**线程安全**的意思是：某个函数（操作）即使被多个线程同时调用，也不会出错、不会产生数据混乱。

### 1.3 什么是"跨线程调用"？

假设服务员 A 负责一楼，服务员 B 负责二楼。

- **同线程调用**：服务员 A 处理一楼的事情——天经地义，没问题。
- **跨线程调用**：服务员 B 跑到一楼去动服务员 A 的东西——可能出问题，需要特别小心。

在程序中，"跨线程调用"就是：在线程 B 里去调用本来属于线程 A 管理的函数或数据。

## 二、muduo 的设计原则

### 2.1 muduo 是什么？

muduo 是一个 C++ 网络编程库，用来写高性能服务器程序。它的作者是陈硕。

### 2.2 muduo 怎么处理线程安全？

muduo 采取了一种**非常清晰的策略**：对于每一个函数，**明确标注**它属于以下哪一类：

| 类别 | 含义 | 类比 |
|------|------|------|
| **线程安全，可跨线程调用** | 任何线程都可以随时调用这个函数，不会出问题 | 餐厅的公共饮水机，谁都可以用 |
| **只能在特定线程调用** | 只有"负责这个事情的线程"才能调用，别的线程不许碰 | 厨房的灶台，只有厨师能操作，服务员不能碰 |

这里的"特定线程"通常是指 **IO 线程**。

---

## 三、什么是 IO 线程？

### 3.1 IO 是什么？

IO = Input/Output（输入/输出）。在网络编程中，IO 就是：

- **收数据**（比如收到用户发来的消息）
- **发数据**（比如给用户发回一条回复）

### 3.2 IO 线程是什么？

muduo 中，每一个 `EventLoop` 对象都运行在一个固定的线程上，这个线程负责处理网络 IO 事件（接收连接、读写数据等），所以叫 **IO 线程**。

关键规则：

> **一个 EventLoop 只属于一个线程，一个线程只运行一个 EventLoop。**

就像一个厨师只能管一个灶台，一个灶台也只能由一个厨师操作。这样就避免了多人同时操作一个灶台导致的混乱。

### 3.3 为什么有些操作只能在 IO 线程执行？

因为 EventLoop 内部管理着网络连接、定时器等资源。如果多个线程同时修改这些资源，就会出现数据竞争（data race），导致程序崩溃或行为异常。

所以 muduo 规定：**操作 EventLoop 内部资源的函数，只能在它所属的 IO 线程中调用。**

## 四、EventLoop 的线程归属检查

### 4.1 问题：怎么防止"用错线程"？

既然规定了"某些函数只能在 IO 线程调用"，那万一程序员不小心在其他线程调用了怎么办？

muduo 的做法是：**在运行时主动检查**，如果发现调用者不在正确的线程上，就立即报错。

### 4.2 两个关键函数

EventLoop 提供了以下两个函数来做这个检查（定义在 EventLoop.h 的第 25～33 行）：

#### `isInLoopThread()` —— 判断是否在正确的线程

```cpp
bool isInLoopThread() const {
    return threadId_ == CurrentThread::tid();
}
```

**通俗解释：**

- `threadId_`：EventLoop 创建时记录下来的"我属于哪个线程"（创建时的线程 ID）
- `CurrentThread::tid()`：当前正在执行代码的线程 ID
- 如果两者相等，说明"对了，是在正确的线程里调用的"，返回 `true`
- 如果不等，说明"错了，是别的线程在调用我"，返回 `false`

> 类比：厨师进厨房前刷一下工牌。系统对比"这个工牌号"和"这个厨房指定的厨师工牌号"，一样就放行，不一样就拦住。

#### `assertInLoopThread()` —— 断言必须在正确的线程

```cpp
void assertInLoopThread() {
    if (!isInLoopThread()) {
        abortNotInLoopThread();  // 不在正确线程就直接终止程序
    }
}
```

**通俗解释：**

- 先调用 `isInLoopThread()` 检查
- 如果检查不通过（在错误的线程调用了），就调用 `abortNotInLoopThread()` **直接让程序崩溃并报错**
- 这是一种"宁可崩溃也不让错误悄悄发生"的防御策略

> 类比：如果一个非厨师闯进了厨房，不是偷偷放行，而是直接拉响警报、关掉厨房，宁可停业也不让非专业人员乱操作。

### 4.3 `abortNotInLoopThread()` 是什么？

原文说"定义从略"，意思是这个函数的具体代码没有展示。但它的功能很简单：

- 打印一条错误日志（告诉你哪个线程在不该调用的地方调用了）
- 调用 `abort()` 终止程序

## 五、整体流程图

```
EventLoop 被创建
      │
      ▼
记录创建它的线程 ID（threadId_）（是成员变量）
      │
      ▼
程序运行中，某个函数被调用
      │
      ▼
调用 assertInLoopThread() 检查
      │
      ├── 当前线程 ID == threadId_？
      │       │
      │       ├── 是 ──→ 正常执行，一切 OK
      │       │
      │       └── 否 ──→ 调用 abortNotInLoopThread()
      │                        │
      │                        ▼
      │                 打印错误日志，程序终止
      │
      └── （保护了程序不会在错误线程上运行）
```

## 六、总结

| 概念 | 一句话解释 |
|------|-----------|
| 线程 | 程序中可以同时跑的多条执行路线 |
| 线程安全 | 多个线程同时调用也不会出错 |
| IO 线程 | 专门处理网络收发数据的线程 |
| EventLoop | 每个 IO 线程的"大管家"，管理该线程上的所有事件 |
| `isInLoopThread()` | 检查"现在是不是在正确的线程上" |
| `assertInLoopThread()` | 检查 + 如果不对就直接崩溃报错 |
| muduo 的策略 | 明确标注每个函数该在哪个线程调用，运行时主动检查，错了就崩溃 |

**核心思想：与其让 bug 悄悄发生、事后难以排查，不如在第一时间崩溃报错，让问题暴露在开发阶段。**
